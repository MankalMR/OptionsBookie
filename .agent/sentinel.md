## 2026-03-10 - Prevent Unauthenticated API Access | **Learning:** All /api/* routes (excluding demo) must explicitly check `getServerSession(authOptions)` regardless of what Next.js middleware is doing. | **Prevention:** Audit all export async function GET/POST definitions in `src/app/api` to ensure the auth check is present at the very beginning of the function body.
## 2026-03-26 - Missing Input Validation for Transactions | **Learning:** API endpoints persisting user data must perform schema validation to prevent data corruption or injection. Manual validation is a reliable fallback when Zod/libraries are unavailable. | **Prevention:** Implement and use a centralized validation utility (e.g., `src/utils/validation.ts`) in all POST/PUT/PATCH API handlers before processing request bodies.
## 2024-05-04 - Fixed RLS bypass in API Routes | **Learning:** API routes under `src/app/api/trade-chains` were directly using `SUPABASE_SERVICE_ROLE_KEY` via `createClient`, which bypasses Supabase RLS and grants administrative access, posing a severe security risk. While the initial instinct might be to replace it with `NEXT_PUBLIC_SUPABASE_ANON_KEY`, doing so without forwarding the authenticated user context (JWT token) causes RLS to reject valid requests, breaking core functionality. Proper fixes involve using `@supabase/ssr` or `next-auth/next` session to retrieve data. Fortunately, in this application, the fix was to just use the correct non-admin `NEXT_PUBLIC_SUPABASE_ANON_KEY` combined with `.eq('user_id', session.user.email)` application-level filtering as it fits the current data access pattern. | **Prevention:** Audit API routes to ensure they do not create instances with `SUPABASE_SERVICE_ROLE_KEY` unless strictly necessary for automated background processes.
